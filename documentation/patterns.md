## Общая архитектура системы

Система реализована на Django и моделирует полный жизненный цикл доставки: от создания заказа до его вручения получателю. Все паттерны работают на реальных объектах базы данных и связаны между собой.

**Модели данных:**
- `User` — пользователь (клиент или курьер)
- `Order` — заказ с тарифом, адресами, ценой и текущим статусом
- `Payment` — платёж, привязанный к заказу
- `Tracking` — запись трекинга
- `Courier` — курьер с рейтингом и флагом доступности
- `RouteGraph` — рёбра графа маршрутов (узел A → узел B, расстояние)

**Жизненный цикл заказа** управляется паттерном State:

```
new → payment_pending → courier_assigned → in_delivery → delivered
                                                        ↘
                                               (из любого состояния) → cancelled
```

**Взаимодействие паттернов:**

```
Abstract Factory   ──создаёт──▶  Order + Payment + Tracking
      │
      └─ вызывает ──▶  Strategy (apply_tariff) ──▶  рассчитывает цену
                              │
Template Method   ──шаги──▶  Strategy + Logger(Singleton) + Courier(ORM)
                              │
         State    ──управляет жизненным циклом Order──▶  next() / cancel()
                              │
        Command   ──оборачивает──▶  State.next() / cancel() / assign_courier
                              │
       Observer   ──уведомляет──▶  Client / Courier / Logger при событии
                              │
        Adapter   ──адаптирует──▶  SMS-шлюз / Email-сервис к Observer
                              │
       Decorator  ──надстраивает──▶  цену и описание поверх Order
                              │
         Proxy    ──контролирует доступ──▶  к изменению статуса / отмене
                              │
         Facade   ──объединяет──▶  Factory + Decorator + Observer + Command
                              │
      Composite   ──строит дерево──▶  заказы по тарифу и статусу
      Iterator    ──обходит──▶  Composite-дерево (DFS), фильтрует листья
      Singleton   ──единый──▶  Logger для всех слоёв системы
      A*/Dijkstra ──ищет──▶  кратчайший маршрут в RouteGraph
```

## 1. Singleton (Одиночка)

**Файл:** `app/patterns/singleton/logger.py`  
**Демо:** `/demo/singleton/`

### Суть

Гарантирует единственный экземпляр класса `Logger` во всём процессе. Реализован через `__new__`: при первом вызове создаётся экземпляр и сохраняется в `_instance`; все последующие вызовы возвращают тот же объект.

```python
class Logger:
    _instance = None

    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance.logs = []
        return cls._instance
```

### Как работает в системе

Все слои системы — Template Method, Abstract Factory, Facade, views — вызывают `Logger()` независимо друг от друга. Так как это всегда один и тот же объект, список `logs` накапливает события со всей системы в едином журнале. На демо-странице наглядно показано, что `id(logger1) == id(logger2)`.

### Зачем это нужно

Без Singleton каждый вызов `Logger()` создавал бы новый пустой журнал, и события от разных слоёв были бы изолированы друг от друга.

## 2. Strategy (Стратегия)

**Файл:** `app/patterns/strategy/tariff_strategy.py`  
**Демо:** `/demo/strategy/`

### Суть

Определяет семейство алгоритмов расчёта цены и делает их взаимозаменяемыми. Абстрактный класс `TariffStrategy` объявляет интерфейс `calculate_price(distance_km: float) -> float`. Три конкретные стратегии реализуют разные формулы:

| Стратегия | Формула | Пример (10 км) |
|---|---|---|
| `EconomyTariff` | `distance × 50` | 500 руб. |
| `StandardTariff` | `distance × 80` | 800 руб. |
| `ExpressTariff` | `distance × 150` | 1 500 руб. |

### Как работает в системе

Метод `Order.apply_tariff()` выбирает стратегию по значению поля `order.tariff` и вызывает `calculate_price(order.distance_km)`. Этот метод вызывается из:
- **Abstract Factory** — сразу после создания заказа
- **Template Method** — на шаге `calculate_price` внутри `execute()`
- **PremiumOrderFactory** — применяет `ExpressTariff`, затем умножает на 1.5

Смена тарифа не требует изменений в логике доставки — достаточно передать другую стратегию.

## 3. State (Состояние)

**Файл:** `app/patterns/state/order_state.py`  
**Демо:** `/demo/state/`, кнопки «Далее» и «Отменить» на странице заказа

### Суть

Позволяет объекту менять поведение при изменении внутреннего состояния. Объект `Order` делегирует методы `next_state()` и `cancel_order()` текущему объекту состояния.

Цепочка переходов:

```
NewOrderState
    → next()  →  PaymentPendingState
                     → next()  →  CourierAssignedState
                                      → next()  →  InDeliveryState
                                                       → next()  →  DeliveredState
    → cancel() →  CancelledState (из любого промежуточного состояния)
```

### Как работает в системе

Текущее состояние восстанавливается «лениво» из поля `Order.status` через свойство `Order.state`:

```python
@property
def state(self):
    if not hasattr(self, '_order_state') or self._order_state is None:
        state_map = {'new': NewOrderState(), 'payment_pending': PaymentPendingState(), ...}
        self._order_state = state_map.get(self.status, NewOrderState())
    return self._order_state
```

При переходе состояние обновляет `order.status`, `order.state_name` и сохраняет запись в БД. Это означает, что состояние переживает перезапуск сервера — оно всегда восстанавливается из базы.

### Зачем это нужно

Без State логика переходов была бы размазана по `if/elif` в нескольких местах. Сейчас каждый статус сам знает, куда можно перейти и что делать при отмене.

## 4. Abstract Factory (Абстрактная фабрика)

**Файл:** `app/patterns/abstract_factory/order_factory.py`  
**Демо:** `/demo/factory/`

### Суть

Создаёт семейства связанных объектов (`Order`, `Payment`, `Tracking`), не привязываясь к конкретным классам. Абстрактная фабрика `OrderComponentFactory` декларирует три метода: `create_order`, `create_payment`, `create_tracking`.

Четыре конкретные фабрики формируют согласованный набор:

| Фабрика | Тариф | Метод оплаты | Статус трекинга | Цена |
|---|---|---|---|---|
| `StandardOrderFactory` | standard | card | «Заказ создан» | по формуле: км × 80 |
| `ExpressOrderFactory` | express | instant | «Экспресс-заказ создан» | по формуле: км × 150 |
| `EconomyOrderFactory` | economy | cash | «Экономный заказ создан» | по формуле: км × 50 |
| `PremiumOrderFactory` | express | card | «Премиум с приоритетным отслеживанием» | km × 150 × 1.5 |

### Как работает в системе

Каждая фабрика при создании заказа вызывает `order.apply_tariff()`, тем самым делегируя расчёт цены паттерну Strategy. `PremiumOrderFactory` дополнительно умножает результат на 1.5, добавляя премиальную наценку. Метод `create_payment` создаёт платёж в БД с нужным способом оплаты; `create_tracking` — запись трекинга с подходящим начальным статусом.

Чтобы добавить новый вид заказа (например, `CorporateOrderFactory`), достаточно реализовать три метода — ни один из существующих файлов менять не нужно.

## 5. Decorator (Декоратор)

**Файл:** `app/patterns/decorator/order_decorator.py`  
**Демо:** `/demo/decorator/`

### Суть

Динамически добавляет поведение к объекту, оборачивая его в декоратор. Базовый `OrderDecorator` хранит ссылку на внутренний объект (заказ или другой декоратор) и делегирует ему вызовы `get_price()` и `get_description()`.

Доступные декораторы:

| Класс | Что добавляет | Надбавка |
|---|---|---|
| `InsuranceDecorator` | Страховка груза | +500 руб. |
| `PriorityDecorator` | Приоритетная доставка | ×1.3 к цене |
| `SMSNotifyDecorator` | SMS-уведомления | +100 руб. |
| `GiftWrapDecorator` | Подарочная упаковка | +300 руб. |
| `FragileDecorator` | Осторожная доставка хрупкого | +800 руб. |
| `SignatureRequiredDecorator` | Подпись при получении | +200 руб. |

### Как работает в системе

Декораторы образуют цепочку: каждый следующий оборачивает предыдущий. Вызов `get_price()` на внешнем декораторе рекурсивно проходит по всей цепочке и накапливает надбавки.

```
Order(400 руб.)
  └─ InsuranceDecorator       → 400 + 500 = 900
       └─ PriorityDecorator   → 900 × 1.3 = 1170
            └─ SMSDecorator   → 1170 + 100 = 1270 руб.
```

Facade использует декораторы автоматически перед запуском процесса доставки.

## 6. Template Method (Шаблонный метод)

**Файл:** `app/patterns/template_method/delivery_process.py`  
**Демо:** `/demo/template-method/`

### Суть

Определяет скелет алгоритма в базовом классе, откладывая реализацию отдельных шагов в подклассы. Базовый класс `BaseDeliveryProcess` реализует метод `execute(order)`, который жёстко задаёт последовательность шагов:

```
validate_order(order)
    → calculate_price(order)   ← общий для всех: вызывает apply_tariff()
        → process_payment(order)
            → assign_courier(order)
                → notify_client(order)   ← общий для всех
```

`validate_order`, `process_payment`, `assign_courier` — абстрактные, каждый подкласс реализует их по-своему:

| Класс | Валидация | Оплата | Назначение курьера |
|---|---|---|---|
| `StandardDelivery` | Проверяет заполненность адресов | `method="card"` | Первый доступный курьер |
| `ExpressDelivery` | Минимальная проверка | `method="instant"` | Курьер с рейтингом ≥ 4.5 |
| `EconomyDelivery` | Базовая проверка | `method="cash"` | Первый доступный курьер |

### Как работает в системе

Общий шаг `calculate_price` вызывает `order.apply_tariff()` — это встроенная точка интеграции со Strategy. На каждом шаге Logger (Singleton) записывает событие. Если любой шаг выбрасывает исключение, `execute()` логирует ошибку и пробрасывает её выше — view поймает и откатит заказ.

## 7. Observer (Наблюдатель)

**Файл:** `app/patterns/observer/order_observer.py`  
**Демо:** `/demo/observer/`

### Суть

Определяет зависимость «один ко многим»: при возникновении события `OrderSubject.notify(order, event)` все подписанные наблюдатели автоматически получают уведомление через `update(order, event)`.

Наблюдатели:

| Класс | Что делает |
|---|---|
| `ClientNotifier` | Формирует сообщение для клиента заказа |
| `CourierNotifier` | Формирует сообщение для назначенного курьера |
| `LoggerObserver` | Записывает событие в собственный лог |

### Как работает в системе

Субъект и наблюдатели полностью независимы: субъект знает только об интерфейсе `OrderObserver`. Наблюдателей можно добавлять и отключать в рантайме через `attach()` / `detach()` без изменения кода субъекта. Observer используется напрямую в `demo_observer`, а также через Facade и Adapter.

## 8. Adapter (Адаптер)

**Файл:** `app/patterns/adapter/notification_adapter.py`  
**Демо:** `/demo/adapter/`

### Суть

Адаптирует несовместимые интерфейсы внешних сервисов уведомлений к интерфейсу `OrderObserver`, принятому в системе. Внешние сервисы (`ExternalSMSGateway.send_message`, `ExternalEmailService.dispatch`) имеют собственные API — адаптеры оборачивают их, реализуя `update(order, event)`.

```
OrderSubject.notify(order, event)
    │
    ├─▶ SMSGatewayAdapter.update()  ──▶  ExternalSMSGateway.send_message(phone, text)
    └─▶ EmailServiceAdapter.update() ──▶  ExternalEmailService.dispatch(email, subject, body)
```

### Как работает в системе

Адаптеры подписываются на `OrderSubject` наравне с обычными наблюдателями — субъект не знает, что за ним стоит внешний сервис. Каждый адаптер собирает результаты отправки в список `sent`, который можно проверить после вызова. Добавление нового канала (например, Telegram) требует только написания нового адаптера — код субъекта не меняется.

## 9. Proxy (Заместитель)

**Файл:** `app/patterns/proxy/order_proxy.py`  
**Демо:** `/demo/proxy/`

### Суть

Предоставляет объект-заместитель, который контролирует доступ к реальному сервису. `OrderServiceProxy` реализует тот же интерфейс `OrderServiceInterface`, что и `RealOrderService`, но перед делегированием проверяет права роли.

Матрица прав:

| Роль | Просмотр | Изменить статус | Отменить |
|---|:---:|:---:|:---:|
| admin | + | + | + |
| courier | + | + | — |
| client | + | — | + |
| guest | + | — | — |

### Как работает в системе

При недостаточных правах прокси выбрасывает `PermissionError` — `RealOrderService` при этом не вызывается вообще. Все обращения (разрешённые и отказанные) фиксируются в `access_log`. Это позволяет видеть полную историю попыток доступа без изменений в реальном сервисе.

## 10. Command (Команда)

**Файл:** `app/patterns/command/order_command.py`  
**Демо:** `/demo/command/`

### Суть

Инкапсулирует запрос как объект, позволяя откатывать операции. Каждая команда реализует `execute()` и `undo()`:

| Команда | `execute()` | `undo()` |
|---|---|---|
| `NextStateCommand` | `order.next_state()` | Восстанавливает предыдущий статус |
| `CancelOrderCommand` | `order.cancel_order()` | Восстанавливает предыдущий статус |
| `AssignCourierCommand` | `order.courier = courier; save()` | Возвращает предыдущего курьера |

### Как работает в системе

`CommandHistory` хранит стек выполненных команд. Вызов `undo()` извлекает последнюю команду из стека и откатывает её действие. Команды используются напрямую в демо, а также в Facade — там `NextStateCommand` переводит новый заказ в `payment_pending` сразу после создания. История команд живёт в памяти процесса в рамках одной сессии.

## 11. Facade (Фасад)

**Файл:** `app/patterns/facade/delivery_facade.py`  
**Демо:** `/demo/facade/`

### Суть

Предоставляет единую точку входа к сложной подсистеме. `DeliverySystemFacade.create_order()` скрывает координацию четырёх паттернов за одним вызовом.

### Как работает в системе

Метод `create_order` выполняет четыре шага последовательно:

1. **Abstract Factory** — создаёт `Order`, `Payment`, `Tracking` через выбранную фабрику (цена рассчитывается через Strategy)
2. **Decorator** — если переданы ключи декораторов, применяет цепочку надбавок и сохраняет итоговую цену
3. **Observer** — уведомляет клиента, курьера и logger о принятии заказа
4. **Command** — через `NextStateCommand` переводит заказ из `new` в `payment_pending`

Каждый шаг добавляет строку в список `steps`, который возвращается в результате — это позволяет видеть, что именно сделала система.

## 12. Composite (Компоновщик)

**Файл:** `app/patterns/composite/order_composite.py`  
**Демо:** `/demo/composite/`

### Суть

Позволяет работать с деревом объектов единообразно. Общий интерфейс `OrderComponent` реализуют два класса:
- `OrderLeaf` — одиночный заказ (лист), всегда возвращает `get_count() == 1`
- `OrderGroup` — группа заказов или подгрупп, суммирует `get_count()` и `get_total_price()` по всем дочерним

### Как работает в системе

Функция `build_tree_from_db()` строит трёхуровневое дерево из реальных данных БД:

```
Все заказы
├─ Стандарт
│    ├─ Новые
│    │    ├─ Заказ #141 — 400.00 руб.
│    │    └─ Заказ #139 — 560.00 руб.
│    └─ В доставке
│         └─ Заказ #135 — 720.00 руб.
├─ Экспресс
│    └─ Доставлены
│         └─ Заказ #122 — 1500.00 руб.
└─ Эконом
     └─ ...
```

Вызов `root.get_total_price()` рекурсивно суммирует цены всех заказов в дереве без знания о структуре вложенности.

## 13. Iterator (Итератор)

**Файл:** `app/patterns/iterator/order_iterator.py`  
**Демо:** `/demo/composite/` (секция итератора)

### Суть

Предоставляет способ последовательного обхода элементов коллекции без раскрытия её внутренней структуры. Реализованы два итератора:
- `OrderIterator` — обходит дерево Composite в глубину (DFS) и собирает все листья
- `FilterIterator` — обёртка над `OrderIterator`, пропускает только заказы с нужным статусом

### Как работает в системе

`OrderIterator` получает корневой узел Composite-дерева и рекурсивно собирает все `OrderLeaf` в плоский список. Оба итератора реализуют `__iter__` и `__next__`, то есть работают в стандартном цикле `for leaf in iterator`. `FilterIterator` при создании проходит по всему `OrderIterator` и фильтрует по `leaf.order.status`. Благодаря такой архитектуре клиентский код (view) не знает ни о структуре дерева, ни о том, как именно производится фильтрация.

## 14. A* / Dijkstra (Кратчайший маршрут)

**Файл:** `app/math_models/shortest_path.py`  
**Демо:** `/demo/astar/`

### Суть

Реализует алгоритм поиска кратчайшего пути в графе маршрутов доставки. Граф хранится в модели `RouteGraph` (рёбра с весом в км) и загружается из БД методом `load_from_db()`. Метод `find_shortest_path(start, end)` возвращает пару `(список узлов маршрута, суммарное расстояние)`.

### Как работает в системе

При открытии демо-страницы граф загружается из `RouteGraph` и строится словарь смежности. Пользователь выбирает начальный и конечный узел — алгоритм находит оптимальный маршрут и возвращает его в виде `A → B → C → D` с суммой расстояний. Это имитирует выбор маршрута для курьера при назначении заказа.
