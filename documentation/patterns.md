## 1. Singleton (Одиночка)

**Файл:** `app/patterns/singleton/logger.py`

Гарантирует, что у класса `Logger` существует только один экземпляр во всей системе.  
Реализовано через переопределение `__new__`: если `_instance` уже создан, возвращается он же.

**В проекте:** единый журнал событий, в который пишут все части системы — смена статусов, создание заказов, ошибки доставки.

---

## 2. Strategy (Стратегия)

**Файл:** `app/patterns/strategy/tariff_strategy.py`

Определяет семейство алгоритмов расчёта цены и делает их взаимозаменяемыми.  
Абстрактный класс `TariffStrategy` задаёт интерфейс `calculate_price(distance_km)`.  
Конкретные стратегии: `EconomyTariff` (50 руб/км), `StandardTariff` (80 руб/км), `ExpressTariff` (150 руб/км).

**В проекте:** метод `Order.apply_tariff()` выбирает стратегию по полю `tariff` и пересчитывает цену.  
Шаблонный метод доставки вызывает `apply_tariff()` — стратегию можно поменять без изменения процесса доставки.

## 3. State (Состояние)

**Файл:** `app/patterns/state/order_state.py`

Позволяет объекту менять поведение при изменении внутреннего состояния.  
`Order` делегирует `next()` и `cancel()` текущему объекту состояния.

Цепочка состояний:  
`NewOrderState` → `PaymentPendingState` → `CourierAssignedState` → `InDeliveryState` → `DeliveredState`  
Из любого состояния (кроме `Delivered`) можно перейти в `CancelledState`.

**В проекте:** жизненный цикл заказа. Состояние восстанавливается из поля `Order.status` при каждом обращении.

## 4. Abstract Factory (Абстрактная фабрика)

**Файл:** `app/patterns/abstract_factory/order_factory.py`

Создаёт семейства связанных объектов (`Order`, `Payment`, `Tracking`) без указания конкретных классов.  
Каждая фабрика (`StandardOrderFactory`, `ExpressOrderFactory`, `EconomyOrderFactory`, `PremiumOrderFactory`) формирует согласованный набор: тариф, метод оплаты, статус трекинга.

**В проекте:** позволяет добавить новый вид заказа (например, `CorporateOrderFactory`), не трогая существующий код.

---

## 5. Decorator (Декоратор)

**Файл:** `app/patterns/decorator/order_decorator.py`

Динамически добавляет обязанности к объекту, оборачивая его в декоратор.  
Базовый класс `OrderDecorator` делегирует `get_price()` и `get_description()` внутреннему объекту.

Доступные декораторы и надбавки:
- `InsuranceDecorator` — +500 руб.
- `PriorityDecorator` — ×1.3
- `SMSNotifyDecorator` — +100 руб.
- `GiftWrapDecorator` — +300 руб.
- `FragileDecorator` — +800 руб.
- `SignatureRequiredDecorator` — +200 руб.

**В проекте:** декораторы можно комбинировать в любом порядке и количестве.

## 6. Template Method (Шаблонный метод)

**Файл:** `app/patterns/template_method/delivery_process.py`

Определяет скелет алгоритма в базовом классе, откладывая реализацию отдельных шагов в подклассы.  
`BaseDeliveryProcess.execute()` вызывает по порядку: `validate_order → calculate_price → process_payment → assign_courier → notify_client`.

Конкретные процессы: `StandardDelivery`, `ExpressDelivery`, `EconomyDelivery` — различаются реализацией абстрактных шагов.

**В проекте:** `calculate_price` использует паттерн Strategy; `Logger` (Singleton) пишет журнал каждого шага.

## 7. Observer (Наблюдатель)

**Файл:** `app/patterns/observer/order_observer.py`

Определяет зависимость «один ко многим»: при изменении состояния субъекта все подписчики автоматически оповещаются.  
`OrderSubject` хранит список `OrderObserver` и вызывает `update(order, event)` при `notify()`.

Наблюдатели:
- `ClientNotifier` — формирует уведомление для клиента
- `CourierNotifier` — формирует уведомление для курьера
- `LoggerObserver` — пишет запись в лог

**В проекте:** субъект и наблюдатели полностью независимы; подписчиков можно добавлять и убирать в рантайме.

## 8. Proxy (Заместитель)

**Файл:** `app/patterns/proxy/order_proxy.py`

Предоставляет объект-заместитель, контролирующий доступ к реальному объекту.  
`OrderServiceProxy` реализует тот же интерфейс `OrderServiceInterface`, что и `RealOrderService`, но перед делегированием проверяет права роли.

Матрица прав:

| Роль    | Просмотр | Изменить статус | Отменить |
|---------|:--------:|:---------------:|:--------:|
| admin   | +        | +               | +        |
| courier | +        | +               | —        |
| client  | +        | —               | +        |
| guest   | +        | —               | —        |

**В проекте:** прокси ведёт `access_log` всех запросов (разрешённых и отказанных).

## 9. Command (Команда)

**Файл:** `app/patterns/command/order_command.py`

Инкапсулирует запрос как объект, позволяя параметризовать клиентов, ставить запросы в очередь и отменять операции.  
Каждая команда реализует `execute()` и `undo()`.

Команды:
- `NextStateCommand` — переход к следующему статусу
- `CancelOrderCommand` — отмена заказа
- `AssignCourierCommand` — назначение курьера

`CommandHistory` хранит стек выполненных команд и предоставляет `undo()`.

**В проекте:** история команд живёт в памяти процесса (в рамках одной сессии); `undo` откатывает последнее действие.

## 10. A* / Dijkstra (Математическая модель)

**Файл:** `app/math_models/shortest_path.py`

Алгоритм поиска кратчайшего пути в графе маршрутов доставки.  
Граф хранится в модели `RouteGraph` (Django ORM), загружается методом `load_from_db()`.  
`find_shortest_path(start, end)` возвращает пару `(список узлов, суммарное расстояние)`.

**В проекте:** используется для оптимального выбора маршрута при назначении курьера.
