## Часть 1. Запуск приложения

### Шаг 1 — Перейти в директорию проекта

Откройте PowerShell или cmd и выполните:

```
cd "C:\путь\до\папки\проекта"
```

Убедитесь, что в папке есть файл `manage.py`:
```
dir manage.py
```
Если файл есть — вы в правильном месте.

### Шаг 2 — Активировать виртуальное окружение

```
sourse\Scripts\activate
```

**Признак успеха:** в начале строки появится `(sourse)`:
```
(sourse) C:\путь\до\проекта>
```

**Если не активируется** (ошибка `...cannot be loaded because running scripts is disabled`):
```
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
```
После этого повторите активацию.

### Шаг 3 — Применить миграции (только при первом запуске или после сброса базы)

```
python manage.py migrate
```

**Ожидаемый вывод:** серия строк вида `Applying app.0001_initial... OK`

Если видите ошибку `no such table` — это значит, что миграции не были применены. Выполните команду ещё раз.

### Шаг 4 — Заполнить базу данных тестовыми данными

```
python populate_db.py
```

**Ожидаемый вывод:**
```
Создано клиентов: 10
Создано курьеров: 7
Создано заказов: 25
Создано маршрутов: 12
```

Скрипт создаст:
- 10 клиентов: логины `client1`–`client10`, пароли `pass1`–`pass10`
- 7 курьеров: логины `courier1`–`courier7`
- 25 заказов с разными тарифами (`economy`, `standard`, `express`) и разными статусами
- 12 рёбер графа маршрутов для алгоритма A* (узлы A–Л)

> **Важно:** если скрипт выполнялся раньше, повторный запуск добавит дубли.  
> Чтобы начать с чистого листа: удалите `db.sqlite3`, затем выполните шаги 3 и 4 заново.

### Шаг 5 — Запустить сервер

```
python manage.py runserver
```

**Ожидаемый вывод:**
```
Starting development server at http://127.0.0.1:8000/
Quit the server with CTRL-BREAK.
```

Откройте браузер: **http://127.0.0.1:8000/**

Оставьте терминал открытым — сервер должен работать всё время демонстрации.

### Шаг 6 — Создать суперпользователя для /admin/ (если ещё не создан)

```
python manage.py createsuperuser
```

Введите логин, email и пароль. Запомните их — они нужны для входа в `/admin/`.

## Часть 5. Типичные вопросы и готовые ответы

| Вопрос | Ответ |
|--------|-------|
| Где паттерн X применяется в реальной системе, не только на демо-странице? | Каждый паттерн задействован в реальных операциях: Facade координирует создание заказа, State управляет жизненным циклом в `/orders/<id>/`, Command пишет историю в журнал |
| Как паттерны связаны между собой? | `DeliverySystemFacade` — точка входа: она использует Factory (создаёт заказ), Decorator (добавляет услуги), Observer (уведомляет), Command (фиксирует действие), Singleton Logger (логирует всё) |
| Почему Proxy, а не просто `if role == 'admin'` в коде? | Логика прав изолирована в отдельном объекте. Добавить новую роль или операцию = изменить только `ROLE_PERMISSIONS` в прокси, не трогать бизнес-логику |
| Что считается математической моделью? | Алгоритм Дейкстры в `app/math_models/shortest_path.py` — используется для расчёта расстояния при назначении курьера |
| Как работает undo в Command? | `CommandHistory` — это стек (`list`). `execute()` кладёт команду через `append`, `undo()` берёт через `pop()`. Каждая команда сохраняет предыдущее состояние перед выполнением |
| Почему Singleton, а не переменная уровня модуля? | Singleton гарантирует единственный экземпляр независимо от того, сколько раз и из скольких модулей вызван конструктор — модульная переменная такой гарантии не даёт |
| Чем Adapter отличается от Decorator? | Adapter меняет интерфейс, не добавляя новое поведение. Decorator добавляет поведение, сохраняя интерфейс |
| Где используется Iterator отдельно от Composite? | `FilterIterator` может обходить любую коллекцию заказов, не только дерево — он не зависит от `OrderGroup` |

```
│   └── EmailServiceAdapter  ← Adapter (ExternalEmailService)
├── Command          → NextStateCommand → CommandHistory (undo)
└── Singleton Logger → логирует все шаги

Order
├── State    → жизненный цикл: New → PaymentPending → CourierAssigned
│              → InDelivery → Delivered | Cancelled
└── Strategy → расчёт цены: Economy(×50) / Standard(×80) / Express(×150)

TemplateMethod (BaseDeliveryProcess.execute)
├── validate_order → calculate_price (Strategy) → process_payment
├── assign_courier → notify_client
└── Singleton Logger

Composite (OrderGroup / OrderLeaf) — оба реализуют get_total_price()
└── Iterator (OrderIterator / FilterIterator) — обход в глубину

Proxy (OrderServiceProxy)
├── ROLE_PERMISSIONS: admin/courier/client/guest
└── RealOrderService → выполняет операцию только при наличии прав

A* (ShortestPathFinder)
└── RouteGraph (таблица БД) → граф маршрутов доставки
```
